## 0. 최우선 순위 원칙

**일정 전제**: 주말에만 작업, 총 1개월 예상. 배포가 긴급하지 않으므로, 아래처럼 "배포 인프라 준비"와 "공개 오픈(+애드센스 신청)"을 분리한다.

- **배포 인프라 준비**(GitHub 브랜치 정리, Vercel 연결, GA 연동)는 1주차에 먼저 끝내둔다. 이후 매주 작업분을 Vercel 미리보기 URL로 실제 확인하며 진행 — 로컬에서만 보는 것보다 안전하고, 마지막에 몰아서 검증할 필요가 없어짐.
- **공개 오픈 + 애드센스 신청**은 Phase 2.5(신규 테스트 4종 + 홈 재설계)까지 끝난 뒤 진행.

**주차별 제안 일정**

| 주차 | 작업 |
|---|---|
| 1주차 | 배포 인프라(Phase 1) + `testEngine.js` 공통 아키텍처 설계 + 테토-에겐 |
| 2주차 | 감다살/감다뒤 (가벼움) + HSP 자애/타애 민감형 (문항 카피 신중하게, 사장님 검토 필요) |
| 3주차 | 성인 애착유형 + 홈 화면 재설계(카테고리 탭, 카드 그리드) |
| 4주차 | 디자인 마감(컬러 토큰, 모션), 공유 기능(OG), 최종 점검 → 공개 오픈 + 애드센스 신청 |

이 순서는 권장안이며, 실제 진행 속도에 따라 클로드 코드가 유동적으로 조정 가능. 다만 **HSP/애착유형처럼 실제 심리검사가 베이스인 콘텐츠는 문항 초안을 반드시 사장님 검토 거친 뒤 확정**하는 원칙은 유지.

## Phase 1 — 배포 전 필수 (개발 작업, 디자인 변경 최소)

이 항목들은 디자인 판단이 필요 없는 순수 개발 작업이라, 별도 스펙 없이 클로드 코드에게 바로 맡겨도 된다.

2. Vercel 배포 (GitHub 연결 → 자동 배포)
3. `NEXT_PUBLIC_SITE_URL` 환경변수 설정 (sitemap이 localhost 기준으로 나가고 있음)

5. `public/eb9da986-....png` (원본 그리드 시트, 2.5MB) 등 코드에서 안 쓰는 용량 큰 파일은 배포 이미지에서 제외 검토

## Phase 2 — 디자인/콘텐츠 스펙 (아래부터가 이 문서의 본론)

### 1. 디자인 시스템

현재 상태: 시스템 폰트, 포인트 컬러 `#2f6fed` 파란색 하나, 라이트/다크 자동 전환만 있음. 나쁘지 않지만 "재미 테스트"치고는 톤이 너무 차분하다 (금융 앱 같은 느낌).

**컬러 토큰 (제안)**
- Primary: `#2f6fed` (기존 파란색 유지 — 이미 여러 곳에 쓰여서 바꾸면 수정 범위가 큼)
- Accent (CTA/포인트 강조용, 신규): `#FF4B6E` (활기찬 코럴핑크 — 버튼 hover, 궁합 점수 숫자, 공유 버튼에만 제한적으로 사용)
- Background: 기존 유지 (라이트 `#fff`, 다크 `#0a0a0a` 계열로 추정)
- 텍스트 컬러 토큰화: 지금 하드코딩된 `#666`, `#aaa`, `#eaeaea` 등을 `globals.css`에 CSS 변수로 선언해서 `page.module.css` 등 여러 파일에서 재사용 (일관성 + 나중에 톤 조정 쉬움)

```css
:root {
  --color-primary: #2f6fed;
  --color-accent: #ff4b6e;
  --color-text-muted: #666;
  --color-border: #eaeaea;
}
@media (prefers-color-scheme: dark) {
  :root {
    --color-text-muted: #aaa;
    --color-border: #2a2a2a;
  }
}
```

**타이포그래피**
- 지금 시스템 폰트로 충분함 (로딩 속도 우선). 폰트 교체는 Phase 3(트래픽 붙은 후)로 미룬다.
- 궁합 점수(`score`)만 크게 강조 — 지금 `.score` 클래스가 얼마나 큰지 코드 기준 불명확했는데, 결과 페이지의 "얼굴"이 되어야 하는 숫자이니 32px 이상 + Accent 컬러 + bold로.

**우선순위**: 컬러 변수화는 낮은 리스크/공수로 전체 톤을 살릴 수 있어서 Phase 1 직후 바로 처리해도 좋음. (CSS 변수로 토큰화하므로 이후 팔레트 조정도 값만 바꾸면 됨 — 리스크 낮음)

**모션/애니메이션 가이드 (신규 — 이미지 추가 대신 UX 힘싣기)**

이미지 에셋을 늘리는 대신, 가벼운 CSS 트랜지션으로 완성도를 높인다. 성능(Core Web Vitals)과 SEO에 영향 없도록 무거운 JS 애니메이션 라이브러리는 쓰지 않고 순수 CSS로 처리한다.

- 결과 점수 카운트업: `/mbti/match/[pair]` 진입 시 궁합 점수가 0에서 실제 점수까지 숫자가 빠르게 올라가는 연출 (결과 "공개 순간"의 임팩트를 살림 — 공유하고 싶은 마음도 커짐)
- 테스트 카드 등장 애니메이션: 홈 화면 테스트 카드 그리드가 순차적으로 살짝 올라오며 fade-in (stagger, 카드당 30~50ms 지연)
- 기존 hover 스케일 효과(`.typeCard:hover`) 유지, 결과 섹션 카드에도 동일한 톤의 hover/tap 효과 확장
- 성별 토글, 카테고리 탭 전환 시 배경색 슬라이드 트랜지션 (지금도 `transition: background 0.15s`는 있음 — 유지하고 다른 토글류에도 일관 적용)
- `prefers-reduced-motion: reduce` 미디어 쿼리 대응 필수 — 모션에 민감한 사용자를 위해 애니메이션 비활성화 옵션 존중

### 2. 바이럴 공유 기능 (신규 — 가장 중요한 추가 기능)

한국 MBTI 테스트 사이트가 트래픽을 얻는 핵심 경로는 **카카오톡 공유 시 미리보기(링크 카드)**다. 지금은 이 부분이 전혀 없다.

**요구사항**
1. `/mbti/match/[pair]` 페이지에 Open Graph 메타 태그 추가
   - `og:title`: 예) "ESTJ × INFP 궁합, 결과는?"
   - `og:description`: 궁합 점수 + 한 줄 요약 (섹션 콘텐츠에서 발췌, 각 궁합마다 자동 생성)
   - `og:image`: 남녀 이미지 페어를 합성한 정적 카드 이미지 (아래 3번 참고)
2. "카카오톡으로 공유하기" 버튼을 결과 페이지 상단에 추가 (Web Share API 우선 사용, 미지원 브라우저는 URL 복사로 폴백 — 카카오 SDK까지는 Phase 1에서 필요 없음, 네이티브 공유로 충분)
3. `og:image`용 결과 카드 이미지는 정적 생성이 어렵다면(136개 조합 x 2 성별 = 272장 사전 렌더링은 과함), Next.js의 `ImageResponse`(OG Image Generation)로 요청 시 동적 생성하는 방식을 클로드 코드에게 검토 요청

**우선순위**: Phase 1 직후 최우선 — 검색 유입 이전에 "공유로 인한 자연 유입"이 더 빠르게 트래픽을 만들어 줄 수 있음.

### 3. 홈 화면 콘텐츠 보강 (SEO/애드센스용)

현재 홈 화면은 제목+한 줄 부제만 있어서 텍스트 양이 매우 적다. 애드센스는 첫 페이지도 심사 대상이다.

**추가 제안**
- 폼 아래, 유형 카드 그리드 위에 3~4문단짜리 소개 섹션 추가: "MBTI 궁합이란", "이 테스트는 어떻게 계산되나요" 같은 설명형 텍스트

**주의**: 텍스트를 늘리는 목적이 "심사 통과"이지 "장문의 벽" 만들기가 아님 — 3~4문단 선에서 멈출 것.

## Phase 2.5 — 멀티 테스트 확장 ("테스트 모음 사이트"로 전환)

사이트 컨셉을 MBTI 단일 사이트에서 **테스트 모음 사이트**로 확장한다. 이유: 콘텐츠 종류가 다양해야 애드센스 심사에 유리하고, 테스트 간 내부 링크로 세션 체류시간이 늘어남. 리서치 결과, MBTI·테토-에겐·HSP(예민형)·애착유형 테스트는 실제로 MZ세대 사이에서 "정체성 테스트"라는 하나의 트렌드 군집으로 함께 소비되고 있어 궁합이 좋다.

**주의**: 이 단계는 MBTI 단독 배포(Phase 1) *이후*에 진행한다. 배포를 늦추는 이유로 쓰지 말 것.

### 2.5.1 아키텍처 변경

- 새 테스트는 `src/app/tests/{slug}/...` 공통 라우트로 추가. 기존 `/mbti/...`는 이미 SEO 인덱싱 대상이 될 URL이라 **손대지 않고 그대로 유지**.
- 데이터 폴더링: `src/data/tests/{slug}/questions.js`, `src/data/tests/{slug}/results.js`로 테스트별 분리.
- 채점 로직 공통화: 문항 응답 → 점수 계산 → 결과 매핑까지의 공통 유틸을 `src/lib/testEngine.js`로 뽑아서 4개 신규 테스트가 재사용.
- 공유(OG 이미지), AdSlot 등 기존 컴포넌트는 테스트 무관하게 재사용 가능하도록 props만 일반화.

### 2.5.2 홈 화면 재설계 (허브형, poomang 레퍼런스 참고)

사장님이 공유한 레퍼런스(poomang) 중 지금 단계에 맞는 요소만 채택한다.

**채택**
- 상단 햄버거 메뉴(☰): 클릭 시 카테고리 목록(`전체 / 성격유형 / 감정심리`)이 드로어로 열림. 데스크톱/모바일 공통으로 이 구조 사용 — 지금의 상단 탭 방식보다 테스트 개수가 늘어나도 확장성 좋음.
- "NEW" 라벨: 최근 추가된 테스트 카드에 뱃지 표시 (신규 테스트 눈에 띄게)
- 테스트 카드 그리드: 정사각형 썸네일 + 제목 + 한 줄 설명 + 예상 소요시간("3분") + 카테고리 pill 태그
- 기존 "나/상대방 MBTI 선택 폼"은 홈 최상단에서 빼고, 카드 그리드 안의 "MBTI 궁합" 카드를 클릭하면 그 폼이 있는 화면으로 이동 (홈이 특정 테스트에 종속되지 않도록)
- 신규 소개 문단(Phase 2의 3번)은 "이 사이트는 무엇인가"로 톤 조정 — MBTI 전용 설명에서 테스트 모음 전체 소개로 업데이트

**채택하지 않음 (지금 단계엔 과함, Phase 3 이후 재검토)**
- 상단 회전형 대형 히어로 배너 — 테스트가 4~5개뿐인 지금은 배너에 넣을 콘텐츠가 부족함. 테스트가 8개 이상 쌓이면 재검토.

**절대 하지 말 것 — 참여 수 뱃지 조작 금지**
- 레퍼런스엔 카드마다 "4801", "5516" 같은 참여자 수가 붙어 있는데, **실제 GA/참여 데이터 없이 숫자를 지어내면 안 된다.** 가짜 사회적 증거는 사용자를 속이는 것이고, 나중에 실제 숫자와 괴리가 생기면 신뢰도 문제가 된다. GA 연동 후 실제 참여 수가 쌓이면 그때 진짜 숫자를 노출한다. 그전까지는 숫자 없이 "NEW" 뱃지나 카테고리 pill만 사용.

### 2.5.3 디자인 — 카테고리 컬러 확장

기존 Primary(`#2f6fed`)/Accent(`#ff4b6e`)에 카테고리 구분용 컬러 하나 추가:

```css
:root {
  --color-category-personality: #2f6fed; /* 성격유형 (MBTI, 테토-에겐) */
  --color-category-emotion: #8b7cf6;     /* 감정심리 (HSP, 애착유형, 감다살) */
}
```

테스트 카드의 카테고리 pill과 결과 페이지 헤더 컬러에 사용. 두 컬러 모두 Accent 코럴핑크와 톤이 겹치지 않도록 확인함.

### 2.5.4 신규 테스트 4종 개요 (문항 상세는 다음 라운드에서 확정)

| 순서 | 테스트 | 카테고리 | 개요 |
|---|---|---|---|
| 1 | 테토-에겐 | 성격유형 | 호르몬 컨셉 2형 분류, 약 12문항 |
| 2 | HSP 자애/타애 민감형 | 감정심리 | 예민도 + 민감 방향(자기/타인) 2축, 학술 HSP-R 문항의 재미용 축약판 |
| 3 | 성인 애착유형 | 감정심리 | 안정/불안/회피/혼란 4유형, ECR 척도 기반 축약판 |
| 4 | 감다살/감다뒤 데일리 체크 | 감정심리 | 5문항 초경량, 신조어 활용, 매일 다른 결과 가능(재방문 유도용) |

**중요 — 신뢰성 관련 주의사항**: HSP와 애착유형은 실제 학술 심리검사(HSP-R, ECR)를 모티프로 한다. 재미 콘텐츠로 가볍게 각색하되, 각 테스트 하단에 "이 테스트는 재미로 즐기는 자가진단이며 전문 심리검사를 대체하지 않습니다" 안내 문구를 반드시 넣는다 — 실제 임상적 진단처럼 오해되지 않도록.
